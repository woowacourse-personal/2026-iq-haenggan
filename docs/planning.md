# 썰풀이 — 서비스 기획서

> **한 줄 정의:** 딱딱한 기사·정보성 글을 넣으면, 인물·갈등·전개가 있는 이야기로 재구성해 "재미있게 읽히고, 하루 뒤에도 기억에 남는" 콘텐츠 변환 플랫폼.
>
> **영감:** 유튜브 '묘하다' — 주식·경제 이야기를 서사로 풀어 통찰을 남기는 방식. 단, 묘하다는 채널이 골라주는 주제만 볼 수 있다. 썰풀이는 **내가 궁금한 그 기사를, 지금, 이야기로** 읽게 한다.

---

## 1. 문제 정의

### 불편
- 경제·주식·시사 기사는 사실 나열이라 맥락이 안 잡히고, 읽어도 금방 잊는다.
- 이야기로 풀어주는 콘텐츠(묘하다류)는 재밌고 기억에 남지만, **크리에이터가 다뤄줄 때까지 기다려야 한다.** 온디맨드가 아니다.
- 즉 "이해되는 형식"과 "내가 원하는 주제" 중 하나를 항상 포기하고 있다.

### 왜 지금인가
- LLM으로 서사 재구성의 한계비용이 0에 가까워짐 → 크리에이터 1명이 하던 걸 온디맨드로 제공 가능해진 첫 시점.
- 뉴스 소비층의 피로: 요약 서비스는 넘치지만 "요약 = 더 빨리 잊는 압축"일 뿐, 이해와 기억의 문제를 풀지 못함.

### 타겟 사용자 (구체적으로)
1. **주 타겟:** 경제·주식에 입문했거나 따라가고 싶은 20~30대. 기사는 어렵고 유튜브 서사 콘텐츠는 즐겨 보는 사람. "금리가 오르면 왜 주가가 빠지는지" 기사로는 이해 못 했지만 이야기로 들으면 이해하는 사람.
2. **부 타겟:** 특정 사건의 전말이 궁금한 사람(기업 분쟁, 정책 변화 등) — 기사 여러 개를 짜맞추기 귀찮은 사람.

### 지금의 대체 행동 (수요의 증거)
- 묘하다·슈카월드류 채널을 구독하고 다뤄주길 기다린다.
- 기사 여러 개를 읽으며 스스로 타임라인을 짜맞춘다.
- 챗봇에 "쉽게 설명해줘"라고 요약을 시킨다 → 쉬워지지만 이야기는 아니라서 기억에 안 남는다.

## 2. 핵심 가치와 차별점

### 요약이 아니라 서사화
| | 요약 서비스 | 썰풀이 |
|---|---|---|
| 하는 일 | 정보를 **줄인다** | 정보를 **이야기 구조로 재배열한다** |
| 결과물 | 짧아진 사실 목록 | 인물·갈등·전개·의미가 있는 서사 |
| 읽고 난 뒤 | 더 빨리 잊음 | 줄거리로 기억에 남음 |
| 측정 | 시간 절약 | **다음 날 회상률** |

### 차별점 3가지
1. **온디맨드 서사화** — 크리에이터를 기다리지 않고, 내가 궁금한 기사를 지금 이야기로.
2. **사실 불변 각색** — 재미를 위해 지어내지 않는다. 사실은 원문에 고정하고 서술 방식만 바꾼다. (§5 파이프라인의 핵심 제약)
3. **통찰 레이어** — 이야기 말미에 "이 사건이 말해주는 것" 한 단락. 묘하다가 사랑받는 진짜 이유는 재미가 아니라 이 통찰이다.

## 3. 핵심 가설 (위험한 순서대로)

프로젝트의 성패를 가르는 가정을 위험한 것부터. 검증도 이 순서로 한다.

- **H1 (가치 가설, 가장 위험):** 같은 기사라도 이야기 형식으로 읽으면, 요약이나 원문보다 **하루 뒤 회상률과 이해도가 유의미하게 높다.**
  - 틀리면: 서비스 존재 이유가 없음. 제일 먼저 검증.
- **H2 (품질 가설):** LLM이 사람 크리에이터 수준은 아니어도, **"원문보다 낫다"고 느낄 수준**의 서사화를 안정적으로 해낼 수 있다.
  - 틀리면: 온디맨드가 불가능, 큐레이션 모델로 후퇴해야 함.
- **H3 (신뢰 가설):** 각색 과정에서 사실 왜곡을 제어할 수 있고, 사용자가 결과물을 신뢰한다.
  - 근거 있는 경계심: 육아 프로젝트 인터뷰에서 실사용자가 "그럴듯하게 틀린 AI 답변"에 강한 거부감을 보임. 같은 리스크가 여기도 있음.
- **H4 (행동 가설):** 사람들이 기사를 만났을 때 "이걸 이야기로 바꿔 읽어야지"라는 **행동을 실제로 한다** (한 번의 신기함이 아니라 반복 사용).

## 4. MVP 범위 — 핵심 기능 딱 하나

> 원칙: "기사 → 이야기 → 읽기". 이것 하나만. 계정, 피드, 추천, 댓글 전부 없음.

### 사용자 흐름
1. 기사 URL 또는 텍스트를 붙여넣는다.
2. 이야기 유형을 고른다 (v1은 2개만: `전말 스토리` / `인물 중심 드라마`).
3. 30초 내 생성된 이야기를 읽는다.
4. 하단에 ① "이 이야기의 사실 근거" 원문 대조 접기(§5의 신뢰 장치), ② 통찰 한 단락.

### MVP에서 뺄 것 (의도적으로)
- 자동 기사 수집/피드 — 온디맨드 검증에 불필요
- 음성(TTS) 버전 — 서사화 가치 검증 후에
- 커뮤니티/공유 — 콜드 스타트 문제를 지금 열 이유 없음
- 다양한 톤 옵션 — 2개 템플릿으로 충분히 H2 검증 가능

## 5. 서사화 파이프라인 (기술 설계)

핵심 난제는 "재미"와 "사실"의 긴장이다. 이걸 한 번의 프롬프트에 맡기지 않고 단계로 분리한다.

```
[기사 원문]
   │
   ▼
① 사실 추출 (Fact Extraction)
   - 인물/기관, 사건, 시점, 수치, 인용을 구조화된 사실 목록으로 추출
   - 이 목록이 이후 모든 단계의 "불변 계약(contract)"
   │
   ▼
② 서사 설계 (Story Arc Mapping)
   - 사실 목록을 이야기 뼈대에 배치: 배경 → 발단(갈등) → 전개 → 절정 → 의미
   - 주인공 선정(기업, 인물, 혹은 '돈의 흐름' 자체)
   │
   ▼
③ 각색 (Narration) — 제약 걸린 생성
   - 규칙: 사실 목록에 없는 사건·수치·발언 생성 금지
   - 허용: 문체, 비유, 장면 전환, 독자에게 말 걸기
   - 금지: 추측을 사실처럼 서술 ("~였을 것이다"는 명시적 추정 표기)
   │
   ▼
④ 사실 대조 (Fact Check Pass)
   - 생성된 이야기의 모든 사실적 주장 ↔ ①의 사실 목록 자동 대조
   - 불일치 발견 시 해당 문장 수정 재생성
   - 산출물: 문장별 근거 매핑 (UI의 "사실 근거 보기"에 사용)
   │
   ▼
[이야기 + 근거 매핑 + 통찰 단락]
```

**설계 포인트**
- ①과 ④가 같은 사실 목록을 공유하는 게 신뢰의 핵심. 각색(③)이 아무리 자유로워도 ④에서 걸러진다.
- 하네스 구성(프롬프트·제약·단계)이 품질을 좌우하므로, **구성 A/B를 나란히 비교하는 실험**으로 개선한다 — 하네스랩에서 고민했던 그 방법론을 여기 적용.
- 비용 최적화: ①·④는 저렴한 모델(또는 로컬 모델)로, ③만 고품질 모델로 — 오프페이에서 고민한 라우팅 구조.

## 6. 검증 로드맵 (프로토타이핑 3회)

> 원칙: 팜파일럿 방식 — 만들 수 있나(기술)보다 **가치가 있나(사람)**를 먼저, 가장 싸게.

### 프로토타이핑 1 — 코딩 0줄, H1(가치) 검증
- **방법:** 최근 기사 3개를 골라 손으로(또는 챗봇 보조로) 이야기 버전 작성. 지인 10명을 두 그룹으로 나눠 A그룹은 원문, B그룹은 이야기 버전을 읽힘.
- **측정:** ① 다 읽었는가(완독률) ② **24시간 뒤 "무슨 내용이었어?" 회상 테스트** ③ "이해됐다" 자가평가
- **성공 기준:** B그룹 회상·이해가 체감될 만큼 우위. 아니면 서사화 방식 자체를 재검토.
- 24시간 인출 테스트는 learning-lab에서 쓰던 방법 그대로라 설계 비용 0.

### 프로토타이핑 2 — 파이프라인 최소 구현, H2·H3(품질·신뢰) 검증
- **방법:** §5 파이프라인을 스크립트 수준으로 구현(웹 UI 불필요, 결과는 문서로 전달). 같은 기사에 대해 [사람 각색 vs LLM 각색]을 블라인드로 비교.
- **측정:** ① "어느 쪽이 더 읽고 싶나" 선호 ② 사실 왜곡 발생 건수(④단계 통과 전/후)
- **성공 기준:** LLM 버전이 원문 대비 명확한 선호를 얻고, 사실 왜곡이 대조 단계에서 잡힘.

### 프로토타이핑 3 — 웹 한 장, H4(행동) 검증
- **방법:** URL 붙여넣기 → 이야기 출력만 되는 웹페이지 배포. 크루·지인 20명에게 뿌리고 **일주일 방치.**
- **측정:** 시켜서가 아니라 **자발적으로 기사를 넣은 횟수**, 재방문 여부, 어떤 기사를 넣는지.
- **성공 기준:** 신기해서 1번 쓰고 끝이 아니라, 뉴스를 만났을 때 여길 떠올리는 사람이 나오는가.

## 7. 리스크와 대응

| 리스크 | 심각도 | 대응 |
|---|---|---|
| **저작권** — 기사 원문 재사용 | 높음 | 원문을 저장·표시하지 않음. 사실(저작권 없음)만 추출해 완전히 새로운 서술 생성. 원문은 링크로만 연결. 인용 최소화. |
| **사실 왜곡** — 각색이 거짓을 만듦 | 높음 | §5의 ④ 사실 대조 단계 + "사실 근거 보기" UI로 투명화. 추정은 추정으로 표기. |
| **투자 조언 오인** — 주식 소재 특성 | 중간 | "이해를 돕는 콘텐츠이며 투자 권유가 아님"을 명시. 종목 매수/매도 표현 생성 금지 제약. 특정 종목 전망 요청은 서사화 거절. |
| **품질 편차** — 기사 유형별 서사화 난이도 차이 | 중간 | v1은 잘 되는 유형(사건·분쟁·정책 전개형)으로 입력을 좁힘. 단순 시황 기사는 "서사화에 적합하지 않음" 안내. |
| **1회성 신기함** — 재사용 안 함 | 중간 | 프로토타이핑 3에서 조기 확인. 실패 시 B2C 툴이 아니라 '크리에이터 보조 도구'로 피봇 여지. |

## 8. 확장 방향 (MVP 검증 후에만)

1. **음성 버전** — 이야기를 TTS로 → 출퇴근 오디오 콘텐츠. (길동무 아이디어와 합류 지점)
2. **시리즈화** — 한 사건의 후속 기사가 나오면 "다음 화"로 이어 붙이는 연재 구조. 사건을 구독한다는 새로운 뉴스 소비.
3. **분야 확장** — 경제 → 과학·기술·판결문·정책 문서. 특히 판결문·공시처럼 원문이 난해한 영역은 서사화 가치가 더 큼.
4. **크리에이터 도구** — B2C가 약하면, 유튜버·블로거가 대본 초안을 만드는 B2B 도구로 전환.

## 9. 성공 지표

- **북극성 지표:** 24시간 회상 테스트 통과율 (이야기 버전 vs 원문) — 서비스의 존재 이유 그 자체
- 보조 지표: 완독률, 자발적 변환 횟수/주, 재방문율, "사실 근거 보기" 클릭률(신뢰 확인 행동)
- 반대 지표(경계): 사실 왜곡 신고 건수 — 1건이라도 나오면 파이프라인 개선이 기능 추가보다 우선

---

## 10. 한 문장 요약

**"요약은 정보를 줄이고, 썰풀이는 정보를 기억에 남게 한다"** — 검증 순서는 가치(회상률) → 품질(LLM 각색) → 신뢰(사실 불변) → 행동(재사용), 그리고 모든 단계에서 사실은 불변 계약이다.
